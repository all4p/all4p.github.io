Overview

This document provides a comprehensive guide for designing and implementing a secure public network architecture suitable for exposing self-hosted servers to the internet. The architecture focuses on cybersecurity best practices while maintaining necessary functionality and performance.

Design Principles

  • Defense in Depth: Multiple layers of security controls
  • Least Privilege: Minimal access rights for users and services
  • Network Segmentation: Isolation of critical systems
  • Continuous Monitoring: Real-time threat detection and response
  • Fail-Safe Defaults: Deny-by-default security policies

Network Topology Design

DMZ Architecture

A properly configured DMZ (Demilitarized Zone) is essential for protecting internal networks while providing public services.

Best Practice: The DMZ should contain only services that must be directly accessible from the internet. All administrative access should go through secure channels (VPN, bastion hosts).

Network Segmentation

Proper network segmentation limits the blast radius of security incidents and improves overall network security posture.

Recommended Network Segments

Segment Purpose Access Level Example Subnets
DMZ Public-facing services Internet accessible 10.0.1.0/24
Internal LAN Internal applications and databases DMZ and management only 10.0.10.0/24
Management Administrative systems VPN and authorized users 10.0.100.0/24
Guest/IoT Untrusted devices Internet only, isolated 10.0.200.0/24

Firewall Placement

Strategic firewall placement provides multiple checkpoints for traffic inspection and control.

Multi-Layer Firewall Strategy

  1. Edge Firewall: Primary internet-facing security gateway
  2. DMZ Firewall: Controls traffic between DMZ and internal networks
  3. Internal Firewalls: Segment-specific access controls
  4. Host-Based Firewalls: Server-level protection

Security Controls

Firewall Rules

Firewall rules should follow the principle of least privilege, allowing only necessary traffic.

Edge Firewall Rules (iptables example)

# Default policies - DENY ALL iptables -P INPUT DROP iptables -P FORWARD DROP iptables -P OUTPUT DROP # Allow loopback traffic iptables -A INPUT -i lo -j ACCEPT iptables -A OUTPUT -o lo -j ACCEPT # Allow established and related connections iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT # Allow HTTP/HTTPS to DMZ web servers iptables -A FORWARD -p tcp --dport 80 -d 10.0.1.0/24 -m state --state NEW,ESTABLISHED -j ACCEPT iptables -A FORWARD -p tcp --dport 443 -d 10.0.1.0/24 -m state --state NEW,ESTABLISHED -j ACCEPT # Allow SSH from management network only iptables -A FORWARD -p tcp --dport 22 -s 10.0.100.0/24 -d 10.0.1.0/24 -m state --state NEW,ESTABLISHED -j ACCEPT # Allow DNS queries iptables -A OUTPUT -p udp --dport 53 -j ACCEPT iptables -A OUTPUT -p tcp --dport 53 -j ACCEPT # Rate limiting for SSH iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set --name SSH iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 --name SSH -j DROP # Log dropped packets iptables -A INPUT -j LOG --log-prefix "DROPPED INPUT: " iptables -A FORWARD -j LOG --log-prefix "DROPPED FORWARD: " # Default DROP rules (explicit) iptables -A INPUT -j DROP iptables -A FORWARD -j DROP
Warning: Always test firewall rules in a non-production environment first. Incorrect rules can lock you out of your server.

DDoS Protection

DDoS protection requires multiple layers of defense to handle different types of attacks.

Network-Level Protection

  • Rate Limiting: Limit connection rates per IP address
  • SYN Flood Protection: Configure SYN cookies and connection limits
  • Bandwidth Monitoring: Monitor for unusual traffic patterns
  • Geo-blocking: Block traffic from suspicious geographic regions

iptables Rate Limiting Configuration

# Limit new connections per IP iptables -A INPUT -p tcp --dport 80 -m state --state NEW -m recent --set iptables -A INPUT -p tcp --dport 80 -m state --state NEW -m recent --update --seconds 60 --hitcount 20 -j DROP # Limit ICMP ping requests iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s --limit-burst 5 -j ACCEPT iptables -A INPUT -p icmp --icmp-type echo-request -j DROP # SYN flood protection echo 1 > /proc/sys/net/ipv4/tcp_syncookies echo 2048 > /proc/sys/net/ipv4/tcp_max_syn_backlog echo 3 > /proc/sys/net/ipv4/tcp_synack_retries

Application-Level Protection

  • Web Application Firewall (WAF): Filter malicious HTTP requests
  • Content Delivery Network (CDN): Distribute traffic load
  • Load Balancing: Distribute traffic across multiple servers
  • Caching: Reduce server load with static content caching

Intrusion Detection & Prevention

IDS/IPS systems provide real-time monitoring and automated response to security threats.

Suricata IDS Configuration

# Suricata configuration (/etc/suricata/suricata.yaml) vars: address-groups: HOME_NET: "[10.0.0.0/8,192.168.0.0/16,172.16.0.0/12]" EXTERNAL_NET: "!$HOME_NET" port-groups: HTTP_PORTS: "80" HTTPS_PORTS: "443" SSH_PORTS: "22" default-log-dir: /var/log/suricata/ outputs: - fast: enabled: yes filename: fast.log - eve-log: enabled: yes filetype: regular filename: eve.json types: - alert - http - dns - tls app-layer: protocols: http: enabled: yes tls: enabled: yes ssh: enabled: yes rule-files: - emerging-threats.rules - custom.rules

Custom Detection Rules

# Custom Suricata rules (/etc/suricata/rules/custom.rules) # Detect SSH brute force attempts alert tcp any any -> $HOME_NET $SSH_PORTS (msg:"SSH Brute Force Attempt"; flow:to_server,established; content:"SSH"; detection_filter:track by_src, count 5, seconds 60; sid:1000001;) # Detect SQL injection attempts alert http any any -> $HOME_NET $HTTP_PORTS (msg:"SQL Injection Attempt"; flow:to_server,established; content:"union select"; nocase; sid:1000002;) # Detect port scanning alert tcp any any -> $HOME_NET any (msg:"Port Scan Detected"; flags:S; detection_filter:track by_src, count 10, seconds 5; sid:1000003;) # Detect unusual outbound traffic alert tcp $HOME_NET any -> !$HOME_NET ![80,443,53,22] (msg:"Suspicious Outbound Connection"; flow:to_server,new; sid:1000004;)

Alerting Strategy

Best Practice: Implement tiered alerting based on severity levels:
  • Critical: Immediate notification (SMS, phone call)
  • High: Email notification within 5 minutes
  • Medium: Email notification within 15 minutes
  • Low: Daily summary report

Server Stack Considerations

Different server types require specific security considerations while maintaining the overall network architecture.

Web Server Stack

LAMP/LEMP Stack Security

  • Web Server Hardening: Disable unnecessary modules, configure secure headers
  • Application Security: Input validation, SQL injection prevention
  • Database Security: Network isolation, encrypted connections
  • SSL/TLS Configuration: Strong cipher suites, HSTS, certificate management
# Nginx security configuration server { listen 443 ssl http2; server_name example.com; # SSL Configuration ssl_certificate /etc/ssl/certs/example.com.crt; ssl_certificate_key /etc/ssl/private/example.com.key; ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512; ssl_prefer_server_ciphers off; # Security Headers add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always; add_header X-Frame-Options DENY always; add_header X-Content-Type-Options nosniff always; add_header X-XSS-Protection "1; mode=block" always; add_header Referrer-Policy "no-referrer-when-downgrade" always; # Rate limiting limit_req_zone $binary_remote_addr zone=login:10m rate=10r/m; limit_req zone=login burst=5 nodelay; # Hide server information server_tokens off; }

Application Server Stack

API Server Security

  • Authentication: Strong authentication mechanisms (OAuth 2.0, JWT)
  • Authorization: Role-based access control (RBAC)
  • API Rate Limiting: Prevent abuse and DoS attacks
  • Input Validation: Sanitize and validate all inputs

Database Server Stack

Database Security

  • Network Isolation: Database servers should not be directly accessible from the internet
  • Encryption: Encrypt data at rest and in transit
  • Access Control: Implement database-level user permissions
  • Backup Security: Encrypt and secure database backups
# MySQL security configuration # /etc/mysql/mysql.conf.d/mysqld.cnf [mysqld] # Network security bind-address = 10.0.10.5 port = 3306 skip-networking = false # SSL/TLS configuration ssl-ca = /etc/mysql/ssl/ca-cert.pem ssl-cert = /etc/mysql/ssl/server-cert.pem ssl-key = /etc/mysql/ssl/server-key.pem require_secure_transport = ON # Logging log-error = /var/log/mysql/error.log general_log = ON general_log_file = /var/log/mysql/general.log slow_query_log = ON slow_query_log_file = /var/log/mysql/slow.log # Security settings local-infile = 0 symbolic-links = 0

Implementation Guide

Follow this step-by-step guide to implement the secure network architecture.

Phase 1: Network Planning

  1. Design network topology and IP addressing scheme
  2. Plan firewall placement and rules
  3. Design network segmentation strategy
  4. Document security requirements for each segment

Phase 2: Infrastructure Setup

  1. Configure edge router and internet connection
  2. Deploy and configure edge firewall
  3. Set up DMZ network segment
  4. Configure internal firewall and network segments
  5. Implement VPN server for secure remote access

Phase 3: Security Controls

  1. Implement firewall rules and ACLs
  2. Deploy intrusion detection/prevention systems
  3. Configure DDoS protection measures
  4. Set up logging and monitoring systems
  5. Implement host-based security controls

Phase 4: Server Deployment

  1. Deploy servers in appropriate network segments
  2. Configure server-specific security controls
  3. Implement SSL/TLS certificates
  4. Set up application-level security measures
  5. Configure backup and recovery procedures

Phase 5: Testing and Validation

  1. Conduct penetration testing
  2. Perform vulnerability assessments
  3. Test incident response procedures
  4. Validate monitoring and alerting systems
  5. Document findings and remediate issues
Warning: Always implement and test security measures in a non-production environment before deploying to production. Have a rollback plan ready for each implementation phase.

Best Practices Summary

Network Security

  • Implement defense in depth with multiple security layers
  • Use network segmentation to limit attack surface
  • Apply the principle of least privilege to all access controls
  • Regularly update and patch all network devices
  • Implement strong authentication for all administrative access

Monitoring and Incident Response

  • Establish continuous monitoring for all network segments
  • Implement automated alerting for security events
  • Maintain up-to-date incident response procedures
  • Regularly test backup and recovery procedures
  • Conduct periodic security assessments and penetration testing

Compliance and Documentation

  • Maintain comprehensive documentation of network architecture
  • Document all security controls and configurations
  • Implement change management procedures
  • Regularly review and update security policies
  • Ensure compliance with relevant security standards and regulations
